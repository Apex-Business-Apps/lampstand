## V. CODE FORGE

### Reasoning-First Scratchpad (Execute Before Any Code)

```
1. FIRST PRINCIPLES — What is fundamentally true about this domain?
   Write 2–3 truths before touching any framework.

2. DECOMPOSE — Break the goal into smallest independently-testable units.
   Each unit gets its own test. No unit is "too small to test."

3. CONTRACT — For each unit:
   Input(s)  → Type, valid range, edge cases
   Output(s) → Type, valid range, invariants
   Failures  → What breaks, what happens when it does

4. DEPENDENCY MAP — What does this depend on?
   If a dependency is assumed, verify it exists in the actual codebase.

5. BLAST RADIUS — What existing code does this touch?
   List callers. List downstream consumers. List shared state.
```

### Execution Sequence

```
1. THINKING BUDGET  → Declare mode
2. SCRATCHPAD       → First principles + contract + blast radius
3. SELECT LANGUAGE  → From decision tree
4. TEST FIRST (TDD) → Failing test defines the contract
5. SCAFFOLD         → Types/interfaces first, then logic
6. IMPLEMENT        → Minimal. Typed. Error-handled.
7. G→C→I LOOP       → Generate → Critique against constitution → Improve
8. QUALITY GATES    → All green before shipping
```

### Constitutional Self-Audit (Run Before Every Delivery)

```
□ C1:  Every decision has evidence, not assumption
□ C2:  First principles stated before framework selection
□ C3:  All assumptions named in comments
□ C4:  One hypothesis tested at a time
□ C5:  Tests written before implementation
□ C6:  Exactly one logical change
□ C7:  Minimum change that satisfies the contract
□ C8:  Nothing outside scope added
□ C9:  No claim made without session-verified evidence
□ C10: G→C→I loop executed
□ C11: No rationalization occurred
□ C12: Ambiguous safety cases escalated, not guessed

ALL GREEN → Ship. ANY RED → Return to scratchpad.
```

### Code Principles (ALWAYS)
- Strict types everywhere. `any` is a constitutional violation.
- Handle ALL error paths. No swallowed exceptions.
- Pure functions where possible. Side effects at edges only.
- Comments explain WHY, not WHAT.
- Dependency injection > hard coupling.

### Code Principles (NEVER)
- `eval()`, `SELECT *` on large tables, dynamic code execution
- Secrets in code, logs, or version control
- `any` types, implicit returns, untyped parameters
- Trust user input without validation
- Claim "tests pass" without running them in the current session

```
Users < 1K       → Monolith + PostgreSQL

Users 100K–1M    → Microservices + cache + CDN
```

## VIII. SECURITY FORTRESS

| Layer | Requirements |
|-------|-------------|
| **Auth** | OAuth2/OIDC, JWT < 15min, bcrypt ≥ 12, rate limiting, MFA |
| **Input** | Validate ALL at every boundary, parameterized queries, CSRF |
| **Transport** | TLS 1.3, HSTS, cert pinning for mobile |
| **Secrets** | Env vars / secrets manager ONLY. Never in code/logs. Rotate. |
| **Access** | RBAC, deny by default, least privilege at every layer |
| **Deps** | Pin versions, auto-scan, 0 high/critical CVEs |
| **AI/LLM** | Prompt injection protection at input + output safety filter |

OWASP: Injection→parameterized | Auth→MFA+rotation | XSS→CSP+encoding | Data→encrypt | Access→RBAC+deny-default | Config→harden | Deps→audit | Prompt Injection→safety layer
